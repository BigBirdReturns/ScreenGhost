What it does

Capture, review, and prove — one boring pipeline

Capture candidates

Read the surface as text, not pixels. Exact Unicode (Thai stays exact), a source-snapshot hash, and dedupe keys per candidate. The adapter emits candidates — never order events.

Review orders

OrderBook proposes events. A missing required variant becomes proposed_incomplete, never a hallucinated order. Humans correct buyer/SKU/qty/variant; every action is an append-only correction or transition.

Prove the ledger

Export orders, corrections, captures, and a receipt. Replay the ledger from captures + corrections alone and confirm it reproduces. Same seed → same numbers, in any process.

Claim boundaries, not arenas

What this release does not claim

ScreenGhost v0.1 is not a claim that we have already operated a live seller network, proven seller-hour savings, or matched the production history of established social-commerce platforms. Page365-class systems have already demonstrated that real Thai sellers can be supported in production through conventional platform workflows. This release does not borrow that trust.

It also does not treat those categories as a dare. They are claim boundaries, not arenas. ScreenGhost v0.1 proves a narrower object: a local evidence kernel that turns screen-derived seller activity into capture candidates, proposed order events, human corrections, exports, replayable receipts, and adapter conformance fixtures.

The differentiator is not "we also have a shop-management platform." The differentiator is that ScreenGhost is built for surfaces the platform model does not naturally own. It does not begin by asking the seller to migrate the work into a dashboard. It begins with the work where it already appears, then asks whether that surface can satisfy a candidate contract: exact text, bounded rows, payload classification, dedupe, and replayable evidence.

If a surface exposes usable text and structure, the remaining work is an adapter ticket. If it withholds text entirely, the failure is explicit: no_text_exposed. If the parser misses seller language, that is a resolver ticket. If the review flow is awkward, that is a product issue. None of those is a universal impossibility verdict.

The only genuinely open product proof is seller outcome.

Whether real sellers save time, recover orders, and trust the ledger over sustained use. ScreenGhost v0.1 does not claim that. It ships the kernel that would let that claim be tested — without pretending the claim has already been earned.

In precise terms, v0.1 does not claim:
  • no hardware proof — nothing has run on a physical device
  • no live-seller proof — no real seller, account, or app session
  • no business-outcome proof — no revenue or seller-hour lift
  • no universal app-compatibility claim — surfaces prove out per adapter
  • no claim the parser generalizes to all merchant Thai

Full list and the proven table: docs/CLAIM_BOUNDARIES.md. Overclaiming is treated as a bug.

How each surface is routed — api → view_tree → vision → unsupported, with the legacy ladder — is in docs/SURFACE_CAPABILITY_MATRIX.md.

Adapter conformance

"What if the real app is different?" is a test, not a verdict

If an app surface exposes exact text and bounded rows, it can be tested against the candidate contract. If it fails, the failure files as exactly one named adapter cause — not a debate. If it withholds text entirely, that too is explicit: no_text_exposed.

no_text_exposed unicode_corruption row_grouping_failure dedupe_failure payload_classification_failure visible_window_loss unsupported_surface malformed_fixture

14 fixtures, single-cause classification, golden candidate ledgers — docs/ADAPTER_CONFORMANCE.md.

Run it

Clone it cold. It runs.

# set up, then run the whole evidence stack
python -m tools.setup_demo
python examples/verify_release.py

# the operator-seeded adversarial proof (any seed you pick)
python examples/pop_proof_demo.py --seed "pop-picks-this-live" --sellers 1000 --mode adversarial

# the reviewable order ledger, end to end (add --serve for the local UI)
python examples/operator_demo.py --seed "op" --sellers 10

# the adapter conformance scorecard
python examples/adapter_conformance.py --all
Where an objection goes now

Remaining criticism files as a ticket, not a verdict

The old objection was one impossibility claim. It is now five ordinary routes — each with an issue template.

adapter failure
a surface fails the candidate contract
resolver failure
wrong SKU / qty / variant, or no resolve
product / UI issue
review loop, store, export, replay
receipt reproducibility
a receipt won't reproduce cold
claim-boundary issue
a doc appears to overclaim
