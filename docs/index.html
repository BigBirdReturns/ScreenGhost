Screen Ghost · the UI is the API

The objection, decomposed.

"The UI can't be the API — Thai is unreliable, vision is slow, payloads are complex, it won't scale, IPs get blocked, users lack infrastructure." That was one impossibility verdict bundling many separable problems. Screen Ghost doesn't argue it. It splits it into categories that are each tested, receipted, or explicitly frozen — so a remaining objection has to arrive as a failing fixture, a resolver case, or a product issue.

1
the durable interface a platform cannot revoke: the one its paying customers use

Not hardware proof. Not live-seller proof. Not business proof. See CLAIM_BOUNDARIES.md.

The bottleneck

Every integration surface except one is revocable

APIs get deprecated, priced, and rate-limited. Reverse-engineered protocols get detected and banned. DOMs churn weekly. Standards need every vendor to cooperate, so they arrive late and partial. But the screen is the product — a platform that breaks its own human interface breaks itself. Anything that operates at the human interface, at human speed, on the official app, cannot be shut out without shutting out the humans.

Why the platform can't win this one

    
The asymmetry flips

    

      Against a protocol adapter, the platform holds every advantage: change the wire format
      Tuesday, the integrator scrambles Wednesday. Against hands on the official app, every
      countermeasure gets crude — and starts injuring their own customers.

How it works

    
Cache meaning, never pixels

    

      A recorded tap-coordinate macro is a per-service adapter rebuilt one macro at a time —
      the Page365 failure mode reborn. Screen Ghost skills store semantic waypoints
      (“tap the element labeled Orders”), resolved against the live screen at
      replay time. Churn costs speed, never capability. And the hands are local-only by design:
      the actuation layer has no network surface, so there is no remote kill switch either.
    

    

      
Canonical intent
“Reply to order #4412” — a stable vocabulary that never churns

      
→

      
Skill fast path
Known label-anchored waypoints, resolved on today's screen

      
→

      
Stale? Re-derive
Label gone → skill demoted → vision model re-reads the new UI

      
→

      
Local hands
Taps on physically attached hardware — no remote surface, no revocation

    

  






  

    
The two objections, answered

    
“It can't read Thai, and it can't scale.”

    

      The strongest case against this design deserves to be stated straight, so here it is:
      a screenshot-and-OCR loop mangles complex scripts and loses the parts that aren't text —
      stickers, locations, attachments, reactions — and a datacenter phone-farm gets fingerprinted
      by its shared origin IP. Both are real. Both are mechanism problems, not laws
      of the approach — and both dissolve the moment you stop treating the screen as an image to OCR
      and stop running the hands in a datacenter.
    


    
Reading fidelity — vision is the fallback, not the front door

    

      
OS view tree
Read the live UI as nodes — the real Unicode string. Thai is exact because it was never a picture. No model on the text path.

      
→

      
Clipboard / copy
Select → copy → read: the exact characters any human could copy off the screen.

      
→

      
Vision, last
Reserved for genuine pixels — a sticker's identity, a map thumbnail — a small, on-demand slice, not the hot path.

    


    
Scale topology — local hands, cloud aggregates

    

      
Where the hands run
The seller's phone
already always-on, already running the app — no PC, no device farm

      
What the cloud does
Aggregate only
moves already-structured events (architecture — hosted scale is a frozen claim, not demonstrated here)

      
Shared origin IPs to fingerprint
0
every action comes from the seller's own device, account, and IP

    

    

      The IP-block objection is the foundation, not the flaw: with no central actuation there is no
      shared origin to detect. The cloud tier carries data, not control — the hands keep no
      network surface, so there is still no remote kill switch.
      


      And because a demo that hides its risks is a boast, not an argument — the two things that can
      still bite: OS accessibility-service policy (expect off-store distribution) and the platform's
      account-ban vector (a consent/product posture, not a capture trick). Full spec, risks carried in
      the open:
      docs/scaling-architecture.md.
